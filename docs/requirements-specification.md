# 分布式秒杀系统需求规约文档

## 一、文档目的

本文档用于描述“分布式秒杀系统”的业务需求、功能需求、非功能需求、角色权限、数据需求和约束条件，为系统设计、开发、测试和验收提供依据。

## 二、项目背景

秒杀系统在短时间内会产生大量读请求和写请求，常见问题包括商品详情访问压力大、库存超卖、重复下单、订单创建拥塞、支付状态不一致、服务被瞬时流量打垮等。

本项目以商城业务为基础，实现商品、库存、订单和秒杀流程，并通过 Redis、Kafka、Nginx、Nacos 等组件演示高并发读写和服务治理能力。

## 三、用户角色

### 1. 普通用户

普通用户可以浏览商品、搜索商品、查看详情、加入购物车、下单、支付、查看物流、签收、评价、追评、领取和使用优惠券、查看消息。

### 2. 管理员

管理员可以管理商品、库存、用户、订单、优惠券、活动、系统通知和秒杀配置。管理员不能执行普通用户的购买、支付、加入购物车等操作。

## 五、非功能需求

### 1. 性能需求

- 商品详情应优先从 Redis 读取，降低数据库压力。
- 秒杀请求应快速返回排队结果，不阻塞等待订单创建完成。
- Nginx 应将请求分发到多个后端实例。

### 2. 一致性需求

- 订单创建和库存扣减必须保持一致。
- 库存不能扣成负数。
- 秒杀失败必须补偿 Redis 库存。
- 支付成功后订单状态必须正确更新。

### 3. 安全需求

- 用户接口需要 JWT 鉴权。
- 管理员接口需要管理员角色。
- 管理员不能访问普通用户购买、支付、购物车等业务能力。

### 4. 可用性需求

- 接口支持限流，避免瞬时流量压垮服务。
- 接口支持熔断，连续异常时快速失败。
- 搜索等非核心路径支持降级保护。

### 5. 可部署需求

- 项目应支持 Docker Compose 一键启动。
- 前端静态资源由 Nginx 托管。
- 后端至少支持两个实例运行。

## 六、数据需求

核心数据包括：

- 用户数据：账号、密码、头像、联系方式、角色。
- 商品数据：名称、价格、图片、分类、销量、秒杀价格、秒杀时间。
- 库存数据：商品库存。
- 订单数据：订单号、用户、金额、状态、收货信息、支付时间。
- 订单明细：商品、数量、单价。
- 评论数据：星级、内容、头像、追评。
- 优惠券数据：券码、门槛、优惠金额、适用商品/品类。
- 秒杀事务日志：请求编号、状态、库存扣减标记、订单号、错误信息。
- 治理配置：限流、熔断、降级参数。

## 七、约束和说明

- 当前项目未接入 MySQL 读写分离。
- 当前项目未接入 Elasticsearch，商品搜索使用 MySQL LIKE。
- 当前项目未接入 ShardingSphere 分库分表。
- 当前项目未接入 Spring Cloud Gateway，使用 Nginx 替代网关入口和负载均衡。

## 八、验收标准

系统满足以下条件即视为核心功能验收通过：

1. 用户可以登录、浏览商品、加入购物车、下单、支付、查看订单。
2. 管理员可以管理商品、库存、秒杀和优惠券。
3. Redis 商品详情缓存生效，并处理穿透、击穿、雪崩。
4. 秒杀接口使用 Redis 预扣库存和 Kafka 异步下单。
5. 数据库库存通过条件更新防超卖。
6. Docker Compose 可以启动完整运行环境。
7. Nginx 可以代理到两个后端实例。
8. 治理接口可以查看限流、熔断状态。
