3 结构设计 3.1 概念结构设计 本数据库反映了“多平台商品比价与动态分享”系统的核心业务模型。主要实体及其关系如下（E-R模型）： 实体：管理员 (Admin) 属性：ID、用户名、加密密码、头像、角色、状态、登录时间等。 实体：用户 (User) 属性：ID、用户名、加密密码、头像、邮箱、注册时间等。 实体：商品 (Product) 属性：商品编号(ID)、名称、描述、当前参考价格、主图、分类、状态等。 实体：动态 (Post) 属性：动态ID、内容、图片、用户ID、点赞数、评论数等。 实体：平台 (Platform/Product_Platform) 属性：关联ID、平台名、平台价格、购买链接等。 实体：价格历史 (Price_History) 属性：记录ID、商品ID、平台、当时价格、记录时间。 实体：评论 (Comment) 属性：评论ID、动态ID、用户ID、内容、评论时间。 实体：点赞 (Like) 属性：点赞ID、动态ID、用户ID、点赞时间。 实体：收藏 (Collection) 属性：收藏ID、动态ID、用户ID、收藏时间。 实体间的主要关系：
关系类型说明
用户 — 动态一对多一个用户可发布多条动态，一条动态只属于一个用户
动态 — 评论一对多一条动态可有多个评论，一个评论只关联一条动态。存在级联删除关系：删除动态将自动删除其所有评论
用户 — 评论/点赞/收藏一对多一个用户可发表多条评论/点赞/收藏
商品 — 商品平台关联一对多一个商品可在多个平台销售，形成比价。存在级联删除关系：删除商品将自动删除其所有平台关联信息
商品平台关联 — 价格历史一对多一个平台上的商品可有多条历史价格记录

3.3 物理结构设计 本部分从系统程序员视角描述物理实现方案。 a. 数据在内存中的安排： 索引区加速：为所有表的主键（id）和外键（user_id, post_id, product_id）创建 B+ Tree 索引以加速表连接。为高频查询的字段如 products.category, products.status, posts.created_at, price_histories 的 (product_id, platform) 复合字段建立了专用索引。对 products 表的 name 和 description 字段建立 FULLTEXT 全文索引以支持高效的商品搜索。 连接池缓冲：通过 mysql2/promise 模块创建连接池，connectionLimit 建议设置为 10，复用数据库连接，减少 TCP 三次握手开销。 b. 外存空间的组织： 存储引擎：全部表统一采用 InnoDB 引擎，支持事务（用于存储过程）、外键约束和行级锁。 存储方式：每张表及其索引存储在一个独立的 *.ibd 表空间文件中。数据和索引由 MySQL 的 InnoDB 引擎自动管理其页（Page）和区（Extent）的分配。 c. 访问数据的方式方法： 参数化查询：后端执行参数化 SQL 语句（如 SELECT * FROM products WHERE id = ?）以彻底防止 SQL 注入。 分页查询：对动态列表等长列表数据，统一采用 LIMIT 和 OFFSET 进行物理分页，确保接口性能。 事务管理：在复杂业务逻辑（如发布动态并更新相关统计）或调用存储过程时，启用数据库事务以保证数据一致性。 4 运用设计 4.1 数据字典设计 本系统的数据字典由详尽的表结构定义和代码中的 SQL 迁移文件共同构成。以下列出核心表的完整数据字典。 4.1.1 管理员表 (admins)

4.2 安全保密设计 数据库设计通过多层机制确保数据安全，具体策略如下： 访问者区分与认证： 区分管理员与普通用户：通过 admins 表和 users 表物理隔离两类账户，其认证逻辑在 API 层有不同的处理。 基于 user_id 的权限控制：对于动态、评论等用户生成内容，API 服务器在执行编辑（UPDATE）和删除（DELETE）操作前，必须验证请求中的 user_id 与数据库中记录的 user_id 是否一致，或请求者是否为管理员。这是应用层的安全保证。 访问类型和数据对象分别对待： 写操作保护：所有创建、更新、删除接口均需验证用户身份。 读操作区分：虽然商品和公开动态默认对所有登录用户可见，但未来可通过 status 字段（如 banned, private）在查询逻辑中过滤数据，实现敏感信息隐藏。 数据完整性保护：通过外键约束和级联删除（CASCADE DELETE） 策略，确保在删除商品或动态时，其关联的平台信息、评论等数据被自动、完整地清理，避免产生“孤儿数据”，从而保证数据结构的整体安全。 数据库层面的防护： 防注入攻击：所有数据库查询均在后端使用预编译参数化语句，从根本上杜绝 SQL 注入的可能性。 敏感数据加密：admins 和 users 表中的 password 字段，在应用程序内部经过哈希算法（如 bcrypt）处理后再入库。数据库中永不存储明文密码。 安全保密设计总览：
安全层面策略实现方式
访问控制区分访问者身份admins 与 users 表物理隔离，API层分别认证
权限验证基于 user_id 的操作控制API层验证 user_id 一致性或管理员身份
数据过滤读写分离与状态过滤通过 status 字段在查询时过滤数据
数据完整性外键与级联删除数据库层 CASCADE DELETE 约束
防注入参数化查询预编译 SQL 语句
密码安全哈希加密存储bcrypt 等算法处理后再入库
